# Plan: Fix Agent Response Truncation and "I received your message" Fallback

## Problem
When the user sends a chat message (e.g., "what do you want to build today?"), the Savant agent responds with "I received your message: \"...\". I'm processing it internally." instead of a substantive response. The reflections panel shows the THINKING section but the actual response is missing or cut off.

## Root Cause Analysis (from code exploration)

### Primary Root Cause
**File:** `crates/agent/src/pulse/heartbeat.rs` (lines ~460-470)

The heartbeat handler has a fallback pattern: when `full_response.trim().is_empty()`, it generates a generic acknowledgment message. This triggers when the agent loop runs but produces no conversational content.

### Contributing Factors

1. **Hidden Tag Over-Filtering** (`crates/agent/src/react/stream.rs` lines ~265-378):
   - The `HIDDEN_TAGS` list includes very generic tags like `"tool"`, `"function"`, `"result"`, `"response"`
   - If the LLM wraps its response in `<thinking>...</thinking>` or `<reasoning>...</reasoning>`, ALL content goes to `full_trace` (telemetry) but NOT to `FinalAnswerChunk` (user-visible)
   - The `clean_answer` stays empty because all content is inside hidden tags

2. **Stream Interruption** (`crates/agent/src/providers/mod.rs` lines ~196-218):
   - When the OpenRouter stream drops (network issue, timeout, provider error), it yields an empty `is_final: true` chunk
   - The agent loop breaks early with empty `full_trace` and `clean_answer`

3. **Tool-Only Loops** (`crates/agent/src/react/stream.rs` lines ~471-753):
   - If the LLM returns only tool calls without text content, the agent loops executing tools
   - When `max_tool_iterations` is reached, a system message is emitted but no conversational response

## Fix Strategy

### Fix 1: Improve Hidden Tag Filtering (stream.rs)
**File:** `crates/agent/src/react/stream.rs`

**Problem:** The `HIDDEN_TAGS` list is too broad. Tags like `"tool"`, `"function"`, `"result"`, `"response"` are too generic and may match content the LLM uses for its actual response structure.

**Fix:** 
- Remove overly generic tags from `HIDDEN_TAGS`: `"tool"`, `"function"`, `"parameter"`, `"arguments"`, `"name"`, `"input"`, `"output"`, `"result"`, `"response"`
- Keep only the specific structural tags that are definitely internal: `"environment_details"`, `"use_mcp_tool"`, `"read_file"`, `"write_to_file"`, `"execute_command"`, `"ask_followup_question"`, `"attempt_completion"`, `"search_files"`, `"list_files"", `"replace_in_file"`, `"browser_action"`, `"mcp_response"`, `"tool_result"`
- For thought tags (`<thinking>`, `<reasoning>`, etc.): emit the content as BOTH a Thought event AND a FinalAnswerChunk so the user sees the response even if it's wrapped in thinking tags

### Fix 2: Improve Fallback Response (heartbeat.rs)
**File:** `crates/agent/src/pulse/heartbeat.rs`

**Problem:** The fallback "I received your message" is unhelpful when the agent genuinely processed the message but had no output.

**Fix:**
- Check if `full_trace` has content (meaning the LLM did respond, but it was all filtered as hidden)
- If `full_trace` has content but `full_response` is empty, use `full_trace` as the response instead of the generic fallback
- Only use the "I received your message" fallback when BOTH `full_response` and `full_trace` are empty

### Fix 3: Handle Stream Interruption Gracefully (providers/mod.rs)
**File:** `crates/agent/src/providers/mod.rs`

**Problem:** Stream interruption yields an empty final chunk, causing the agent loop to think it got a complete but empty response.

**Fix:**
- When a stream interruption occurs, check if any content was accumulated before the interruption
- If content was accumulated, emit it as a final chunk rather than an empty one
- Add a warning log when stream interruption occurs with accumulated content
