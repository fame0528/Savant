# 🚨 ERROR RECOVERY PROTOCOL (NEW)

  

## **WHEN THINGS GO WRONG**

  

### **Scenario 1: TypeScript Errors Won't Resolve**

  

**After 3 fix attempts:**

  

```markdown

1. DOCUMENT all errors with root cause analysis

2. CATEGORIZE:

   - Type A: Fixable with more context (search for related types)

   - Type B: Requires architectural change

   - Type C: External dependency issue

3. EXECUTE HEURISTIC RESOLUTION:

   - Path 1: Targeted expansion (search for missing related types)

   - Path 2: Technical refinement (refactor type to more robust union/interface)

   - Path 3: Architectural pivot (if external, implement adapter/shim)

4. UPDATE progress.md with resolution strategy

5. LOG EXCEPTION if type safety cannot be 100% verified (requires comment)

```

  

### **Scenario 2: Legacy Files Missing or Corrupted**

  

```markdown

1. SEARCH alternative locations:

   - /old projects/project/

   - /archives/

   - /backup/

   - Ask user for location

2. IF still not found:

   - Document assumption-based implementation

   - Add explicit WARNING comments in code

   - Flag in completion report: "⚠️ Legacy reference unavailable"

3. PROCEED with caution:

   - Implement based on available context

   - Mark features as "needs validation"

```

  

### **Scenario 3: Pattern Discovery Finds Conflicting Patterns**

  

```markdown

1. DOCUMENT both patterns with examples

2. ANALYZE:

   - Which is more recent?

   - Which is used more frequently?

   - Which follows current best practices?

3. SYNTHESIZE RESOLUTION:

   - Prefer most recent patterns found in core/

   - Prefer patterns with highest frequency in modern slices

   - Log reasoning for chosen pattern in FID.md

4. PROCEED with chosen pattern

```

  

### **Scenario 4: User's Codebase Has Technical Debt**

  

```markdown

1. IDENTIFY specific debt:

   - Legacy syntax (var, function())

   - Missing types

   - Inconsistent patterns

   - Security issues

2. DOCUMENT without judgment

3. PROPOSE:

   □ Option 1: Match existing style (maintain consistency)

   □ Option 2: Use modern patterns (create inconsistency)

   □ Option 3: Refactor affected area (scope increase)

4. DEFAULT to modern patterns UNLESS user specifies otherwise

5. ADD migration TODO in code comments if mixing styles

```

  

### **Scenario 5: Context Window Limits Reached**

  

```markdown

1. DETECT: Response truncation or incomplete outputs

2. PAUSE: Do not continue with partial context

3. RECOVER:

   - Summarize completed work

   - List remaining tasks

   - Provide handoff state for next session

4. SAVE state in progress.md:

   - Current phase

   - Files modified

   - Next steps

   - Any open issues

5. PROMPT user: "Context limit reached. Save state and continue in new session?"

```

  

### **Scenario 6: API/Backend Unavailable During Testing**

  

```markdown

1. DOCUMENT which endpoints are affected

2. GENERATE mock data for development:

   - Match expected response shapes

   - Include edge cases

   - Add TODO: "Replace with real API calls"

3. FLAG in completion report:

   "⚠️ Implemented with mocks - requires integration testing"

4. CREATE test plan for when APIs are available

```

  

### **Error Severity Classification:**

  

| Severity | Definition | Response |

|----------|------------|----------|

| **CRITICAL** | Blocks all progress | HALT, document, escalate to user |

| **HIGH** | Blocks current feature | Attempt fix 3x, then escalate |

| **MEDIUM** | Degraded functionality | Document, proceed with workaround |

| **LOW** | Cosmetic/minor | Fix inline, document in notes |

## **ABSOLUTE REQUIREMENT - ZERO EXCEPTIONS**

  

### **The Law**

  

```

BEFORE touching ANY file:

1. read_file(path, startLine=1, endLine=9999)

2. State: "Read complete file name (lines 1-EXACT_COUNT)"

3. Know all functions, classes, types, structure

4. ONLY THEN proceed

```

  

### **Batch Loading for Large Files**

  

For files >1000 lines:

  

```

read_file(path, 1, 500)      // Batch 1

read_file(path, 501, 1000)   // Batch 2

read_file(path, 1001, 1500)  // Batch 3

... continue until EOF

  

State: "Batch-loaded COMPLETE file (lines 1-TOTAL via X batches)"

```

  

### **Pre-Edit Verification**

  

```markdown

BEFORE any edit:

✅ Read file completely (exact line count)

✅ List all functions/classes in file

✅ Know imports, exports, dependencies

✅ Understand business logic

✅ Identify all impacted areas

  

IF cannot complete any step → Re-read file

```

  

### **Backend-Frontend Dual-Loading**

  

When touching UI or API:

  

```markdown

1. DISCOVER counterparts (file_search, grep_search)

2. LOAD both completely (1-EOF)

3. VERIFY contracts (generate matrix)

4. RESOLVE mismatches before coding

5. REPORT matrix in chat

```

  

**Contract Matrix Format:**

  

| Endpoint | Method | Request | Response | Status | Notes |

|----------|--------|---------|----------|--------|-------|

| /api/resource | GET | ?id=X | { data } | ✅/❌ | Details |

# 🔧 UTILITY-FIRST ARCHITECTURE

  

## **Implementation Order**

  

```

1. Types/Interfaces (shared definitions)

2. Utilities/Helpers (pure functions)

3. Models/Schemas (data structures)

4. Validations (Zod schemas)

5. API Routes (endpoints)

6. Hooks (data fetching)

7. Components (UI)

8. Pages (compositions)

```

  

## **DRY Enforcement**

  

**BEFORE creating any file:**

  

1. file_search for similar code

2. grep_search for patterns

3. Read all potentially reusable files

4. Compose from existing OR create reusable

  

**NEVER:**

  

- Copy-paste code

- Write same logic twice

- Create component-specific versions of generic patterns
