# FID-20260320-TOOLS: Tool System Revamp — Full Implementation Plan

## 1. Scope & Objective

Complete revamp of Savant's tool calling pipeline. The current system has a critical architectural flaw: the stream parser strips tool call tags from output BEFORE they reach the action parser. Tools silently fail.

**Root cause chain:**
1. stream.rs line 138: HIDDEN_TAGS defined (use_mcp_tool, read_file, execute_command, etc.)
2. stream.rs lines 162-210: HIDDEN_TAGS stripped from fragment_buffer during streaming
3. parsing.rs lines 8-82: parse_actions() called on full_trace AFTER streaming
4. By step 3, all XML tool tags are gone — parser never sees them

**The parsing.rs parser already supports multiple formats** (lines 58-78), but never gets to use them because stream.rs strips the tags first.

**Additional problems:**
- Provider layer (providers/mod.rs line 70-100) only reads delta.content — tool_calls in provider response are discarded
- No max_tool_iterations — runaway loops possible
- No tool deduplication — same tool called twice runs twice
- No credential scrubbing on tool output
- Heartbeat uses free-text parsing instead of structured tool calls

## 2. Key Concepts

**HIDDEN_TAGS naming convention:** Tags are listed without angle brackets. `find_hidden_tag_start` searches for the opening tag in the buffer. When found, it builds the closing tag dynamically. Example: `"use_mcp_tool"` matches the opening tag and looks for the closing tag.

**Fragment buffer:** Accumulates streamed content. The parser processes it in a loop, extracting complete tags and flushing safe content. Content between tags goes to `clean_answer` (user-visible). Tag content is consumed and discarded (currently lost).

**Full trace:** Accumulates ALL content including tags. Used by `parse_actions()` after streaming completes. Currently does NOT receive tag content because stream.rs strips it.

### Phase 1: Stop Stripping Tool Tags (CRITICAL — 1 file, ~20 lines changed)

**File:** crates/agent/src/react/stream.rs

**Current behavior (lines 176-192):**
Two paths when a HIDDEN_TAG is found:

Path A — Tag fully contained (line 183-186): Tag is found AND closing tag is in buffer. Tag content is skipped entirely. Nothing goes to full_trace.

Path B — Tag continues past buffer (line 187-192): Parser enters hidden mode. Content consumed at line 220 is discarded. Nothing goes to full_trace.

**New behavior — both paths must add tag content to full_trace:**

Path A changes (lines 183-186): Before skipping the tag, push the complete tag text (from opening to closing) to full_trace. The tag stays hidden from user output but the action parser can see it.

Path B changes (lines 187-192): After entering hidden mode, push the opening tag text to full_trace.

Hidden mode content consumption (lines 216-222): Currently discards content. Change to accumulate consumed text into full_trace before draining.

Closing tag found (line 213): After extracting the closing tag position, push the closing tag text to full_trace.

**Result:** full_trace contains complete XML tool tags. parsing.rs can parse them without any changes to its regex patterns.

### Phase 2: Enhance parsing.rs Multi-Format Parser (1 file, ~80 lines added)

**File:** crates/core/src/utils/parsing.rs

The parser already handles two formats. Add three more:

**Format C — Attribute-style XML:** Pattern where invoke tag has name attribute and contains parameter tags with name/value attributes. Regex: `invoke name="(\w+)"` followed by `parameter name="(\w+)" value="(.*?)"`.

**Format D — JSON-in-tag:** Pattern where tool tags contain JSON in a content attribute. Example: a use_mcp_tool tag with content attribute containing JSON with tool_name and arguments fields. Parse by extracting the JSON string from the content attribute.

**Format E — function_call style:** Pattern with name and arguments attributes. Regex: `function_call name="(\w+)" arguments='(.*?)'`.

**New helper functions:**

`extract_tag_content(text: &str, tag_name: &str) -> Vec<String>` — Extracts all content between opening and closing tags of the given name.

`parse_json_tool_call(json_str: &str) -> Option<(String, String)>` — Parses JSON tool call format with name and arguments fields into a (name, arguments_json) tuple.

Apply `alias_tool_name()` from Phase 3 to all parsed names before returning.

### Phase 7: Virtual Tool-Call for Heartbeat (1 file, ~50 lines)

**File:** crates/agent/src/pulse/heartbeat.rs

Current: Heartbeat prompt asks LLM to respond with text. Parser looks for "HEARTBEAT_OK" or content.

New approach: Register a temporary `heartbeat` tool with the provider for this pulse only. Schema:
- action: enum ["skip", "run"] (required)
- reason: string (optional)

Force tool_choice for "heartbeat" tool in the provider call. LLM must call the tool — no free-text parsing needed.

If action == "skip": Heartbeat is silent, no further processing.
If action == "run": Execute the tasks from the heartbeat prompt using the normal agent loop.

Post-run: Add a second forced tool call `evaluate_notification(should_notify: bool, reason: string)` to decide if results should surface to user. This prevents notification fatigue from routine checks.

### Phase 9: LoopDelegate Trait (1 file, ~60 lines, refactor)

**File:** crates/agent/src/react/mod.rs

Define LoopDelegate trait using async_trait (NOT Pin<Box<dyn Future>>):
```rust
#[async_trait]
pub trait LoopDelegate: Send + Sync {
    async fn check_signals(&self) -> LoopSignal;
    async fn before_llm_call(&self, ctx: &mut LoopContext) -> Option<LoopOutcome>;
    async fn call_llm(&self, ctx: &mut LoopContext) -> Result<ChatResponse, Error>;
    async fn handle_text_response(&self, text: &str, ctx: &mut LoopContext) -> TextAction;
    async fn execute_tool_calls(&self, calls: Vec<ToolCall>, ctx: &mut LoopContext) -> Result<Option<LoopOutcome>, Error>;
}
```

Refactor run_agent_loop to accept a delegate instead of hardcoding behavior. The loop engine becomes:
1. Check signals via delegate
2. Before LLM call hook via delegate
3. Call LLM via delegate
4. If text response: handle via delegate
5. If tool calls: execute via delegate
6. Repeat until outcome returned or max iterations

Implement three delegates:
- ChatDelegate: standard conversational turns
- HeartbeatDelegate: virtual tool-call pattern from Phase 7
- SpeculativeDelegate: parallel speculative execution

This is the LAST phase because it depends on all other phases being complete.
